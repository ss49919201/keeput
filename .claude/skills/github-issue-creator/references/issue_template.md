# GitHub Issue テンプレートガイド

## 標準的なIssue構造

keeputリポジトリの既存Issueに基づいて、以下の構造に従う：

### タイトル形式
- 明確で、アクション指向の説明
- 動詞で始める（例: 「追加」「修正」「実装」）
- 何の機能/コンポーネントが変更されるかを含める

### 本文形式

```markdown
[目的の簡潔な1〜2文の概要]

## 仕様
- [仕様のポイント1]
- [仕様のポイント2]
- [仕様のポイント3]

## [コンテキストに基づくオプションセクション]
### 計算方法
```
[該当する場合の計算例]
```

### 出力例
```
[出力例]
```

### 判定基準
```
[該当する場合の基準]
```

## 関連
[関連ドキュメントへの参照、例: "analyzer詳細分析機能の一部（plan.md参照）"]
```

## リポジトリの実例

### 例1: シンプルな機能追加
```
タイトル: Otel Resource のサービス名はアプリケーションで定義

本文:
サービス名は固定し、環境名を環境変数で定義できるようにする。
```

### 例2: 仕様を含む機能
```
タイトル: 最新アウトプット選出時に複数プラットフォームを考慮する

本文:
複数プラットフォームの中から選出する。
優先順は既存の仕様を維持する。
```

### 例3: 詳細な技術的機能
```
タイトル: 週次/月次の記事公開数を集計する機能を追加

本文:
過去7日間および過去30日間の記事公開数を集計し、執筆活動量を定量的に把握できるようにする。

## 仕様
- 過去7日間の記事公開数を集計
- 過去30日間の記事公開数を集計
- 複数プラットフォーム（Zenn, Hatena）の記事を合算

## 出力例
```
PublishedLast7Days: 2   // 過去7日間で2件公開
PublishedLast30Days: 8  // 過去30日間で8件公開
```

## 関連
analyzer詳細分析機能の一部（plan.md参照）
```

## 重要な原則

1. **簡潔性**: 説明は簡潔だが情報量のあるものにする
2. **明確性**: 具体的な例と仕様を使用する
3. **構造**: 適切な場合はセクション（## 仕様、## 出力例など）を使用
4. **参照**: 関連するドキュメントやIssueにリンクする
5. **言語**: すべての内容に日本語を使用する
